# Module A：AI 基础 - 从 LLM 到 Agent Workflow

## 1. LLM 基本工作方式

**核心原理**：基于上下文进行概率生成。给定一段文本，预测最合理的下一个 token 序列。

**能做什么**：
- 语言理解与生成
- 代码生成与解释
- 逻辑推理与分析
- 多语言翻译

**不能做什么**：
- 精确事实记忆（会"幻觉"）
- 确定性计算（数学可能出错）
- 跨会话状态保持（默认无记忆）
- 获取实时信息（知识有截止日期）

## 2. 四个控制层面

| 层面 | 作用 | 类比 |
|------|------|------|
| 上下文窗口 | 模型能看到多少信息 | 工作内存/桌面大小 |
| 系统指令 | 设置身份、语气和边界 | 岗位说明书 |
| 提示词 | 传递当前任务意图 | 本次工单 |
| 工具调用 | 让模型从"说话"变为"做事" | 手和脚 |

## 3. LLM API 调用核心参数

```python
response = client.chat.completions.create(
    model="gpt-4o",           # 选用哪个模型
    messages=[                 # 对话历史
        {"role": "system", "content": "你是一个助手"},
        {"role": "user", "content": "你好"}
    ],
    temperature=0.7,           # 0=确定性, 1=创造性
    max_tokens=1000            # 最大输出长度
)
```

**主要平台**：
- OpenAI：GPT-4o / o3 / o4-mini
- Anthropic：Claude Opus / Sonnet / Haiku
- Z.ai / GLM：GLM-4 系列，OpenAI 兼容接口

## 4. Prompt → Workflow → Agent 边界

| 类型 | 决策者 | 路径 | 失控风险 | 可调试性 |
|------|--------|------|----------|----------|
| Prompt | 人 | 单次问答 | 低 | 高 |
| Workflow | 预定义 | 固定流程 | 中 | 高 |
| Agent | 模型自主 | 动态规划 | 高 | 低 |

**什么时候用什么**：
- 一次性问答 → Prompt
- 流程固定 → Workflow / 脚本
- 目标开放、需多工具协作 → Agent
- 高合规要求 → 人工审核节点

## 5. AI Coding 工具的价值与限制

**能加速的**：
- 样板代码生成
- 陌生库的快速理解
- 原型搭建
- Bug 诊断与修复建议

**不能替代的**：
- 代码审查（安全、性能、可维护性）
- 测试设计（边界条件、集成测试）
- 架构决策（技术选型、系统设计）
- 业务逻辑正确性验证

## 6. AI 输出为什么必须验证

| 风险类型 | 表现 | 应对 |
|----------|------|------|
| 事实错误 | 自信编造不存在的信息 | 关键事实须外部核实 |
| 引用错误 | 编造论文、链接、数据来源 | 不信任模型给的链接 |
| 推理漂移 | 长上下文中逻辑链断裂 | 分段验证 |
| 执行越权 | Agent 超出授权范围操作 | 设置 guardrails + human-in-the-loop |
| 工具误用 | 调用错误工具或参数 | 用 tracing 监控 |

## 7. Agent 核心技术组件

- **状态管理**：多节点共享读写同一 State
- **长期记忆**：跨 session 存储与召回
- **MCP**：LLM 与外部工具的统一连接协议
- **Skills**：可复用的高层指令集
- **Tool Calling**：模型输出结构化请求，框架执行后回传结果
- **Tracing**：可视化 agent 执行链
- **Guardrails**：输入输出验证规则
- **Handoff**：子任务完成后移交控制权
- **错误恢复**：重试、回退和人工介入
