# Module B：Web3 基础 - 账户、钱包、签名与链上执行

## 1. 账户、地址和钱包的关系

```
助记词 (12/24 words)
    ↓ 推导
私钥 (256-bit)
    ↓ 椭圆曲线算法
公钥
    ↓ Keccak-256 哈希
地址 (0x...)
```

**关键概念**：
- **钱包** ≠ 普通账号。钱包是私钥管理器 + 链上动作入口
- **地址** = 公开身份，可查余额和交易历史，但不等于匿名
- **私钥** = 控制权。谁持有私钥，谁就能控制对应地址的资产
- **助记词** = 私钥的可读备份。泄露 = 失去所有资产

**安全底线**：
- 助记词和私钥永不泄露、不截图、不上传
- AI Agent 不应直接接触私钥/助记词
- 涉及签名、授权、转账、合约写入必须保留人工确认

## 2. 签名与交易

**签名 ≠ "点一下确认"**

签名是在授权一个具体动作：
- 转账：授权从 A 地址转 X 代币到 B 地址
- 合约调用：授权执行某个合约函数
- 授权（Approve）：允许某地址使用你的代币

**交易生命周期**：
```
构造交易 → 签名 → 广播到网络 → 进入 mempool → 矿工/验证者打包 → 区块确认 → 完成
```

## 3. Gas 机制

**Gas 是什么**：链上执行的计算成本单位。

**为什么需要 Gas**：
- 防止无限循环（合约执行有上限）
- 激励验证者（谁帮你执行交易，谁获得 Gas 费）
- 网络资源定价（越拥堵越贵）

**Gas 计算**：
```
交易费 = Gas Used × Gas Price
Gas Used：实际消耗的计算步骤
Gas Price：每步的价格（由网络拥堵程度决定）
```

**L1 vs L2**：
- L1（如以太坊主网）：安全性最高，Gas 最贵
- L2（如 Arbitrum、Optimism）：继承 L1 安全性，Gas 便宜 10-100 倍

## 4. 智能合约

**与普通后端逻辑的差异**：

| 特性 | 智能合约 | 普通后端 |
|------|----------|----------|
| 状态 | 公开可查 | 私有数据库 |
| 执行 | 公开透明 | 服务器内部 |
| 升级 | 需权限检查/可能不可更改 | 随时可改 |
| 成本 | 每次执行消耗 Gas | 服务器费用 |
| 确定性 | 必须确定性执行 | 可以有随机性 |

**最小合约示例（Solidity）**：
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract SimpleStorage {
    uint256 private storedValue;

    function set(uint256 x) public {
        storedValue = x;
    }

    function get() public view returns (uint256) {
        return storedValue;
    }
}
```

## 6. 区块浏览器

**Etherscan（etherscan.io）能做什么**：
- 查看交易详情（哈希、状态、Gas、区块高度）
- 查看地址余额和交易历史
- 查看合约代码和状态
- 验证合约源码

**交易回执关键字段**：
- TxHash：交易唯一标识
- Status：Success / Fail
- Block：所在区块高度
- Gas Used / Gas Price：实际消耗
- From / To：发送方和接收方

## 7. 高级延展：AI × Web3 关键基础设施

**账户抽象（Account Abstraction / ERC-4337）**：
- 让合约账户也能发起交易（不只是 EOA）
- 支持社交恢复、批量交易、Gas 代付
- 为 AI Agent 提供可编程的权限控制

**智能账户 / 多签（Safe）**：
- 多个私钥共同控制一个账户
- 设置阈值（如 3/5 签名才能执行）
- AI Agent 可以作为其中一个签名者，但不拥有完全控制权

**为什么这些对 AI × Web3 重要**：
- AI Agent 需要有限权限（不能全权控制资产）
- 需要可审计的执行记录
- 需要失败恢复机制
- 需要人工确认关键操作
