# 概念串联：从 LLM 到链上执行

## 一句话版本

> 用户对 AI 说"帮我部署一个合约" → LLM 理解意图并生成代码 → 人审查代码 → 钱包签名交易 → 链上执行 → 区块浏览器验证结果。

## 逐个概念解释

### LLM（大语言模型）

**是什么**：基于海量文本训练的概率生成模型，给定输入预测最合理的下一个 token。

**在这条链路中的角色**：理解用户意图，生成 Solidity 代码、交易脚本或操作说明。

**局限**：可能生成有漏洞的代码，不能保证编译通过，不了解链上实时状态。

### Prompt（提示词）

**是什么**：传递给模型的文本输入，包含任务指令、上下文和约束。

**在这条链路中的角色**：告诉 AI "用 Solidity 写一个存储合约"或"解释这笔交易的 input data"。

**关键**：Prompt 质量决定输出质量。模糊的指令 → 模糊的代码。

### Workflow（工作流）

**是什么**：预定义的任务流程，每一步的输入输出和路径固定。

**在这条链路中的角色**：
```
用户提需求 → AI 生成代码 → 人工审查 → 编译测试 → 部署 → 验证
```
每一步有明确的前置条件和输出，不依赖 AI 自主决策。

### Agent（智能体）

**是什么**：能自主规划、调用工具、管理状态的 AI 系统。

**与 Workflow 的区别**：Agent 能根据中间结果动态调整策略（如部署失败时自动修复代码重试），Workflow 路径固定。

**在这条链路中的潜在角色**：自动监控链上状态、触发合约交互、管理多步操作。但 Week 1 我们用的是 Workflow 模式，人控制每一步。

### Tool Use（工具调用）

**是什么**：LLM 输出结构化请求，框架执行后回传结果。

**在这条链路中的角色**：AI 调用编译器编译代码、调用 RPC 查询链上状态、调用钱包发起交易。

**关键**：AI 决定"要做什么"，工具负责"怎么做"。

### 钱包（Wallet）

**是什么**：私钥管理器 + 链上动作入口。不是银行账户，不存储代币。

**在这条链路中的角色**：持有私钥，对交易进行签名，授权链上操作。

**安全边界**：AI 不应直接接触私钥。签名前必须人工确认。

### 签名（Signature）

**是什么**：用私钥对交易数据进行加密授权。

**在这条链路中的角色**：授权"将这段合约代码部署到链上"这个具体动作。

**关键**：签名 = 授权，不是"点确认"。一旦广播，不可撤销。

### 交易（Transaction）

**是什么**：经过签名的状态变更指令，广播到网络后被打包进区块。

**在这条链路中的角色**：携带合约字节码，请求网络创建合约实例。

**生命周期**：构造 → 签名 → 广播 → 进入 mempool → 打包 → 确认。

### Gas

**是什么**：链上计算的成本单位。

**在这条链路中的角色**：合约部署消耗 199056 Gas（你的 SimpleStorage），比普通转账的 21000 贵 9 倍，因为需要存储代码。

**为什么需要**：防止滥用，激励验证者，定价计算资源。

### 智能合约（Smart Contract）

**是什么**：部署在链上的可执行代码，状态公开、执行透明。

**在这条链路中的角色**：AI 生成的 Solidity 代码被编译、部署到链上，成为可交互的合约实例。

**与后端的区别**：部署后难以修改，每次调用消耗 Gas，所有人都能查看代码和状态。

### 测试网（Testnet）

**是什么**：功能与主网相同但代币无价值的网络，用于学习和测试。

**在这条链路中的角色**：我们在 Sepolia 测试网完成所有操作，不涉及真实资产。

### 区块浏览器（Block Explorer）

**是什么**：链上数据的只读查询工具。

**在这条链路中的角色**：验证交易状态、查看合约代码、确认 Gas 消耗、追溯操作历史。

## 完整链路图

```
用户："帮我部署一个存储合约"
        ↓
   ┌─────────────┐
   │  LLM 理解意图  │ ← Prompt 决定质量
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │ AI 生成 Solidity │ ← Tool Use: 调用代码生成
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │  人工审查代码  │ ← 不能跳过！AI 可能生成漏洞
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │  编译 & 测试   │ ← Workflow: 固定步骤
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │  钱包签名交易  │ ← 私钥授权，人工确认
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │  链上执行部署  │ ← Gas 消耗，不可撤销
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │  区块浏览器验证 │ ← 确认交易状态和合约地址
   └─────────────┘
```

## 关键边界

| 环节 | 谁负责 | 能否自动化 |
|------|--------|-----------|
| 理解意图 | AI | 可以 |
| 生成代码 | AI | 可以 |
| 审查代码 | 人 | 不应自动化 |
| 编译测试 | 工具 | 可以 |
| 签名交易 | 人 + 钱包 | 不应自动化（除非有严格 guardrails） |
| 链上执行 | 网络 | 自动 |
| 验证结果 | 人 + 浏览器 | 可辅助，但人做最终判断 |
