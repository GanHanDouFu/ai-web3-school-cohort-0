# Module C：最小交叉实验 - AI 输出到链上执行

## 实验目标

把 AI 与 Web3 放到同一条任务链上，体验：
AI 输出 → 人工复核 → 钱包确认 → 链上执行 → 区块浏览器验证

## 实验流程图

```
┌─────────────┐
│  1. AI 生成  │  Claude Code 生成合约代码/交互脚本
└──────┬──────┘
       ↓
┌─────────────┐
│ 2. 人工复核  │  检查代码逻辑、安全性、参数正确性
└──────┬──────┘
       ↓
┌─────────────┐
│ 3. 钱包确认  │  MetaMask 弹窗，确认交易参数
└──────┬──────┘
       ↓
┌─────────────┐
│ 4. 链上执行  │  交易广播到测试网
└──────┬──────┘
       ↓
┌─────────────┐
│ 5. 区块验证  │  在 Etherscan 查看交易结果
└─────────────┘
```

## 实验方案

### 方案 A：AI 生成合约 → 测试网部署

1. 用 Claude Code 生成一个 Solidity 合约
2. 人工审查合约逻辑和安全性
3. 在 Remix IDE 中编译和部署到 Sepolia 测试网
4. 在 Etherscan 验证部署结果
5. 调用合约函数，记录交易哈希

### 方案 B：AI 解释交易 → 学习记录

1. 找一笔真实的链上交易
2. 用 Claude Code 解释交易的 input data
3. 人工核实解释的准确性
4. 整理成学习笔记

### 方案 C：AI 生成任务计划 → 链上执行

1. 让 AI 生成一个 Web3 学习任务计划
2. 标记哪些步骤涉及链上操作
3. 对链上操作步骤设置人工确认点
4. 执行并记录每个步骤的结果

## 关键边界与风险

| 环节 | 风险 | 应对措施 |
|------|------|----------|
| AI 生成 | 代码有漏洞/逻辑错误 | 必须人工审查 |
| AI 生成 | 引用不存在的函数/库 | 验证编译通过 |
| 钱包确认 | 误操作确认恶意交易 | 仔细核对 To/Value/Data |
| 链上执行 | 测试网交易失败 | 检查 Gas、参数、网络 |
| 验证阶段 | 误读浏览器数据 | 核对 TxHash 和 Status |

## 为什么需要这条完整链路

1. **AI 不可信但有用**：生成速度快，但必须验证
2. **链上不可撤销**：一旦执行无法回滚，必须在执行前确认
3. **人是最终决策者**：AI 是工具，人负责判断和确认
4. **可审计性**：每一步都有记录，可追溯和复盘
