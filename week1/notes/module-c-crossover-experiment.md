# Module C：最小交叉实验 - AI 输出到链上执行

## 为什么要做这个实验

AI 和 Web3 各自有一套工作方式。这个实验的目的是第一次把它们放到同一条链路上，看看：
- AI 输出的东西能不能直接用？
- 哪些环节必须人工介入？
- 链上执行和 AI 生成之间怎么衔接？

## 实验流程

```
AI 生成 → 人工复核 → 钱包确认 → 链上执行 → 区块浏览器验证
```

每一步都有明确的输入、输出和检查点。

## 我实际做的实验

### 实验 1：AI 生成合约 → 测试网部署

**过程**：
1. 告诉 Claude Code："帮我写一个最简单的 Solidity 存储合约"
2. Claude Code 生成了 SimpleStorage.sol
3. 我检查了代码逻辑（set/get 函数、owner 设置）
4. 在 Remix 编译通过
5. 部署到 Sepolia 测试网
6. 调用 set(42)，再调用 get() 验证返回 42

**AI 做了什么**：
- 生成了完整的合约代码
- 包含了事件（ValueChanged）和 owner 权限
- 代码可直接编译部署

**我人工做了什么**：
- 检查了函数可见性（public vs external）
- 确认没有重入漏洞（这个简单合约没有外部调用）
- 在 Remix 中手动编译验证

**发现的问题**：
- AI 生成的代码没有使用 SafeMath（不过 Solidity 0.8+ 内置了溢出保护，所以没问题）
- 如果是复杂合约，AI 可能遗漏安全检查

### 实验 2：AI 解释交易 → 学习记录

**过程**：
1. 我的合约部署交易：`0x1d334ce3...`
2. 让 Claude Code 解释这笔交易的 input data
3. Claude Code 解释了字节码的结构（部署代码 + 运行时代码 + 构造函数）
4. 我在 Etherscan 上核对了关键字段

**AI 做了什么**：
- 解释了合约创建交易和普通转账的区别
- 说明了 Gas 差异的原因
- 解释了合约地址的生成方式

**我人工做了什么**：
- 在 Etherscan 核对了 From、To、Value、Gas Used
- 确认"合约创建交易 To 为空"这个说法正确
- 验证了合约地址确实存在

**发现的问题**：
- AI 解释的字节码结构是通用知识，没有实际反编译我的合约
- Gas 估算的细节可能因编译器版本而异

## 关键边界与风险

| 环节 | 风险 | 我的应对 |
|------|------|----------|
| AI 生成合约 | 有漏洞/逻辑错误 | 必须编译+人工审查 |
| AI 引用库 | 可能不存在/版本错 | 在 Remix 实际编译验证 |
| 钱包确认 | 误点确认 | 仔细核对 To/Value |
| 链上执行 | 失败 | 检查余额和 Gas |
| 验证结果 | 看错数据 | 核对 TxHash 和 Status |

## 为什么这条链路重要

1. **AI 快但不可信**：生成代码几秒钟，但可能有漏洞
2. **链上不可撤销**：部署了就改不了（除非用代理模式）
3. **人是最后一道防线**：AI 是工具，人负责判断
4. **每步可追溯**：交易哈希、区块高度、Etherscan 链接，全部可查

## 我的反思

> 这个实验让我理解了 AI × Web3 的核心矛盾：AI 擅长快速生成，Web3 要求精确无误。两者结合的关键不是让 AI 全自动，而是在正确的环节插入人工检查。
>
> 具体来说：
> - 代码生成：AI 做，人审查
> - 编译测试：工具做，人确认
> - 签名交易：人做（AI 不应接触私钥）
> - 验证结果：人做（AI 可辅助解读）
