# Module A：AI 基础 - 从 LLM 到 Agent Workflow

## 1. LLM 到底在做什么

### 核心原理

LLM 的本质是一个"下一个词预测器"。给它前面的文字，它算出接下来最可能出现的词是什么。

**类比**：就像手机输入法的联想功能，但强大几万倍。输入法只能联想几个字，LLM 能联想出一整篇文章。

**具体过程**：
```
输入："今天天气"
  ↓ 模型计算
候选："真" (概率 35%) | "不" (概率 20%) | "很" (概率 18%) | ...
  ↓ 选择"真"
输入："今天天气真"
  ↓ 模型计算
候选："好" (概率 40%) | "热" (概率 25%) | "冷" (概率 15%) | ...
  ↓ 选择"好"
输出："今天天气真好"
```

### 能做什么 vs 不能做什么

| 能做 | 不能做 | 为什么 |
|------|--------|--------|
| 写文章、翻译 | 记住精确事实 | 它记的是"模式"不是"数据" |
| 写代码、解释代码 | 做精确数学计算 | 概率生成 ≠ 计算器 |
| 分析问题、推理 | 保持跨会话记忆 | 每次对话从零开始 |
| 理解上下文 | 获取实时信息 | 知识有截止日期 |

### 关键理解

**"幻觉"不是 bug，是特性**。LLM 的设计目标就是"生成最合理的下一个词"，不是"说真话"。它会自信地编造看起来合理但实际不存在的信息。所以：
- 问它"法国首都是哪"→ 大概率对（训练数据里太多次了）
- 问它"某个小众论文的 DOI"→ 很可能编造

## 2. 四个控制层面

这四个层面决定了 LLM 的行为，理解它们才能有效使用 AI。

### 上下文窗口 = 工作台大小

模型一次能"看到"多少文字。就像你的桌面，桌面越大，能摊开的资料越多。

- GPT-4o：128K tokens（约 10 万字）
- Claude：200K tokens（约 15 万字）

**实际影响**：如果对话太长，前面的内容会被"遗忘"（超出窗口）。

### 系统指令 = 岗位说明书

告诉模型"你是谁"、"你该怎么做"。比如：
```
你是一个 Solidity 审计专家。检查代码时重点关注重入攻击和溢出漏洞。
```
系统指令设定了模型的行为边界，用户看不到但始终生效。

### 提示词 = 本次工单

你每次发给模型的具体问题或任务。提示词的质量直接决定输出质量。

**好的提示词**：
```
用 Solidity 写一个 ERC-20 代币合约，要求：
1. 有 mint 和 burn 函数
2. mint 只有 owner 能调用
3. 添加溢出保护
```

**差的提示词**：
```
写个代币合约
```

### 工具调用 = 手和脚

让模型从"说话"变成"做事"。模型可以调用搜索、计算、代码执行等工具。

```
用户："今天北京天气怎么样？"
  ↓ 模型决定需要查天气
模型输出：调用 weather_api(city="北京")
  ↓ 工具返回结果
模型输出："北京今天晴，25°C"
```

## 3. Prompt → Workflow → Agent

这三个概念是递进关系，理解边界才能选对方案。

### Prompt：一问一答

```
用户 → 提问 → 模型回答 → 结束
```
决策权在人。模型只负责回答这一次问题。

**适合**：翻译、解释、写一段代码、回答一个问题。

### Workflow：固定流程

```
步骤1 → 步骤2 → 步骤3 → 输出
```
每个步骤的输入输出和路径都是预定义的。模型可能是其中某个步骤，但不能改变流程。

**适合**：数据处理流水线、CI/CD、固定格式的报告生成。

**例子**：
```
收到邮件 → AI 提取关键信息 → 写入数据库 → 发送确认邮件
```

### Agent：自主决策

```
目标 → 模型规划步骤 → 执行 → 根据结果调整 → 继续 → 完成
```
模型自己决定下一步做什么、调用什么工具、什么时候停止。

**适合**：开放式问题、需要多工具协作、中间结果决定下一步策略。

**例子**：
```
用户："帮我调查这个 bug"
Agent → 读日志 → 发现异常 → 搜索相关代码 → 定位问题 → 提出修复方案
```

### 选错方案的代价

| 场景 | 用错了 | 后果 |
|------|--------|------|
| 简单问答用了 Agent | 过度工程 | 浪费资源，增加出错可能 |
| 复杂任务用了 Prompt | 不够用 | 输出质量差，需要大量人工补充 |
| 需要人工确认的用了全自动 Agent | 越权操作 | 可能造成不可逆损失 |

## 4. AI Coding 工具

### 我的使用体验（Claude Code）

**做得好的**：
- 能读取整个项目结构，生成的代码风格一致
- 能直接执行 git 操作，从生成到提交一条龙
- 能记住之前的对话上下文

**需要警惕的**：
- 生成的 Web3 相关代码可能有安全漏洞
- 引用的库版本可能过时
- 复杂逻辑可能有边界条件遗漏

### AI Coding 的正确定位

AI 是**加速器**，不是**替代者**。

- ✅ 让 AI 写样板代码，人审查逻辑
- ✅ 让 AI 解释陌生代码，人判断正确性
- ✅ 让 AI 生成测试用例，人补充边界条件
- ❌ 直接部署 AI 生成的合约代码到主网
- ❌ 不审查就 merge AI 生成的 PR
- ❌ 让 AI 独自做架构决策

## 5. AI 输出验证清单

每次使用 AI 输出前，过一遍这个清单：

- [ ] **事实核查**：关键事实是否可从权威来源验证？
- [ ] **引用核查**：论文、链接、数据来源是否存在？
- [ ] **逻辑检查**：推理链是否完整？结论是否跟前提一致？
- [ ] **权限检查**：AI 做的操作是否在授权范围内？
- [ ] **工具检查**：调用的工具和参数是否正确？
- [ ] **代码审查**：是否有安全漏洞？边界条件是否处理？

## 6. Agent 核心技术组件

### 逐个理解

**状态管理**：
Agent 在多步操作中需要记住中间结果。比如"查到了日志中的错误行号"这个信息，后面的"定位代码"步骤需要用到。

**长期记忆**：
跨对话的记忆。比如"这个用户喜欢简洁的代码风格"，下次对话时自动应用。

**MCP（Model Context Protocol）**：
让 AI 能连接外部工具的统一协议。就像 USB 接口，有了它，AI 能接各种"外设"（数据库、API、文件系统）。

**Tool Calling**：
AI 输出结构化请求，框架执行后回传结果。
```
AI: 我需要搜索代码 → 输出 {"tool": "grep", "query": "function transfer"}
框架: 执行搜索 → 返回结果
AI: 根据结果继续分析...
```

**Tracing**：
记录 Agent 每一步做了什么，方便调试。就像程序的日志系统。

**Guardrails**：
输入输出的验证规则。比如"不允许 Agent 调用 rm -rf"、"输出必须包含免责声明"。

## 7. 我的学习笔记

### 我的理解

> LLM 就是一个超级强大的"文字接龙"机器。它不理解语义，但通过海量数据学会了语言的模式。所以它能写出看起来很有道理的话，但可能完全是编的。

### 我的疑问

> - Agent 的"自主决策"边界在哪？什么时候该让人介入？
> - 长期记忆会不会导致 Agent 记住错误信息？
> - MCP 和传统的 API 调用有什么本质区别？

### 我的发现

> 使用 Claude Code 时，提示词越具体，输出质量越高。比如"帮我写一个 Web3 概念测验"不如"帮我写一个 10 道选择题的 Web3 概念测验，深色主题，每题有解释，纯 HTML/CSS/JS"。
