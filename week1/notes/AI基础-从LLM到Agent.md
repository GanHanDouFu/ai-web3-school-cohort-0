# Module A：AI 基础 - 从 LLM 到 Agent Workflow

## 1. LLM 到底在做什么

### 核心原理

LLM 的本质是一个"下一个词预测器"。给它前面的文字，它算出接下来最可能出现的词是什么。

**类比**：就像手机输入法的联想功能，但强大几万倍。输入法只能联想几个字，LLM 能联想出一整篇文章。

**具体过程**：
```
输入："今天天气"
  ↓ 模型计算
候选："真" (概率 35%) | "不" (概率 20%) | "很" (概率 18%) | ...
  ↓ 选择"真"
输入："今天天气真"
  ↓ 模型计算
候选："好" (概率 40%) | "热" (概率 25%) | "冷" (概率 15%) | ...
  ↓ 选择"好"
输出："今天天气真好"
```

### 能做什么 vs 不能做什么

| 能做 | 不能做 | 为什么 |
|------|--------|--------|
| 写文章、翻译 | 记住精确事实 | 它记的是"模式"不是"数据" |
| 写代码、解释代码 | 做精确数学计算 | 概率生成 ≠ 计算器 |
| 分析问题、推理 | 保持跨会话记忆 | 每次对话从零开始 |
| 理解上下文 | 获取实时信息 | 知识有截止日期 |

### 关键理解

**"幻觉"不是 bug，是特性**。LLM 的设计目标就是"生成最合理的下一个词"，不是"说真话"。它会自信地编造看起来合理但实际不存在的信息。所以：
- 问它"法国首都是哪"→ 大概率对（训练数据里太多次了）
- 问它"某个小众论文的 DOI"→ 很可能编造

## 2. 四个控制层面

这四个层面决定了 LLM 的行为，理解它们才能有效使用 AI。

### 上下文窗口 = 工作台大小

模型一次能"看到"多少文字。就像你的桌面，桌面越大，能摊开的资料越多。

- GPT-4o：128K tokens（约 10 万字）
- Claude：200K tokens（约 15 万字）

**实际影响**：如果对话太长，前面的内容会被"遗忘"（超出窗口）。

### 系统指令 = 岗位说明书

告诉模型"你是谁"、"你该怎么做"。比如：
```
你是一个 Solidity 审计专家。检查代码时重点关注重入攻击和溢出漏洞。
```
系统指令设定了模型的行为边界，用户看不到但始终生效。

### 提示词 = 本次工单

你每次发给模型的具体问题或任务。提示词的质量直接决定输出质量。

**好的提示词**：
```
用 Solidity 写一个 ERC-20 代币合约，要求：
1. 有 mint 和 burn 函数
2. mint 只有 owner 能调用
3. 添加溢出保护
```

**差的提示词**：
```
写个代币合约
```

### 工具调用 = 手和脚

让模型从"说话"变成"做事"。模型可以调用搜索、计算、代码执行等工具。

```
用户："今天北京天气怎么样？"
  ↓ 模型决定需要查天气
模型输出：调用 weather_api(city="北京")
  ↓ 工具返回结果
模型输出："北京今天晴，25°C"
```

## 3. Prompt → Workflow → Agent

这三个概念是递进关系，理解边界才能选对方案。

### Prompt：一问一答

```
用户 → 提问 → 模型回答 → 结束
```
决策权在人。模型只负责回答这一次问题。

**适合**：翻译、解释、写一段代码、回答一个问题。

### Workflow：固定流程

```
步骤1 → 步骤2 → 步骤3 → 输出
```
每个步骤的输入输出和路径都是预定义的。模型可能是其中某个步骤，但不能改变流程。

**适合**：数据处理流水线、CI/CD、固定格式的报告生成。

**例子**：
```
收到邮件 → AI 提取关键信息 → 写入数据库 → 发送确认邮件
```

### Agent：自主决策

```
目标 → 模型规划步骤 → 执行 → 根据结果调整 → 继续 → 完成
```
模型自己决定下一步做什么、调用什么工具、什么时候停止。

**适合**：开放式问题、需要多工具协作、中间结果决定下一步策略。

**例子**：
```
用户："帮我调查这个 bug"
Agent → 读日志 → 发现异常 → 搜索相关代码 → 定位问题 → 提出修复方案
```

### 选错方案的代价

| 场景 | 用错了 | 后果 |
|------|--------|------|
| 简单问答用了 Agent | 过度工程 | 浪费资源，增加出错可能 |
| 复杂任务用了 Prompt | 不够用 | 输出质量差，需要大量人工补充 |
| 需要人工确认的用了全自动 Agent | 越权操作 | 可能造成不可逆损失 |

## 4. AI Coding 工具

### 我的使用体验（Claude Code）

**做得好的**：
- 能读取整个项目结构，生成的代码风格一致
- 能直接执行 git 操作，从生成到提交一条龙
- 能记住之前的对话上下文

**需要警惕的**：
- 生成的 Web3 相关代码可能有安全漏洞
- 引用的库版本可能过时
- 复杂逻辑可能有边界条件遗漏

### AI Coding 的正确定位

AI 是**加速器**，不是**替代者**。

- ✅ 让 AI 写样板代码，人审查逻辑
- ✅ 让 AI 解释陌生代码，人判断正确性
- ✅ 让 AI 生成测试用例，人补充边界条件
- ❌ 直接部署 AI 生成的合约代码到主网
- ❌ 不审查就 merge AI 生成的 PR
- ❌ 让 AI 独自做架构决策

## 5. AI 输出验证清单

每次使用 AI 输出前，过一遍这个清单：

- [ ] **事实核查**：关键事实是否可从权威来源验证？
- [ ] **引用核查**：论文、链接、数据来源是否存在？
- [ ] **逻辑检查**：推理链是否完整？结论是否跟前提一致？
- [ ] **权限检查**：AI 做的操作是否在授权范围内？
- [ ] **工具检查**：调用的工具和参数是否正确？
- [ ] **代码审查**：是否有安全漏洞？边界条件是否处理？

## 6. Agent 核心技术组件（详细）

### 状态管理（State Management）

**是什么**：Agent 在多步操作中，多个节点共享读写同一份状态数据。

**类比**：就像多人协作的白板。每个人（节点）都能看到白板上的内容，也能往上写东西。后面的人能看到前面人写的结果。

**为什么重要**：没有状态管理，Agent 每一步都是孤立的。比如：
- 第 1 步：搜索代码，发现错误在第 42 行
- 第 2 步：需要修复第 42 行的代码
- 如果没有状态，第 2 步不知道"42 行"这个信息

**我在 Claude Code 中的体验**：
- Claude Code 能记住整个对话上下文（相当于同一个 session 内的状态）
- 比如我说"帮我写一个合约"，然后说"加上事件"，它知道"合约"指的是刚才写的那个
- 但如果开一个新对话，它就不记得了（状态不跨 session）

**实际例子**：
```
用户："帮我检查这个文件的安全问题"
Agent State = { file: "contract.sol", issues: [] }
  ↓ 第 1 步：读取文件
Agent State = { file: "contract.sol", content: "...", issues: [] }
  ↓ 第 2 步：分析代码，发现重入漏洞
Agent State = { file: "contract.sol", content: "...", issues: ["重入漏洞在第15行"] }
  ↓ 第 3 步：生成修复方案（引用了 issues 中的信息）
输出："第 15 行存在重入漏洞，建议使用 ReentrancyGuard"
```

---

### 长期记忆（Long-term Memory）

**是什么**：跨 session 的记忆存储与召回。下次对话时能自动应用之前学到的信息。

**类比**：短期记忆 = 黑板（擦掉就没了），长期记忆 = 笔记本（随时翻看）。

**为什么重要**：没有长期记忆，每次对话都从零开始。用户需要反复说明自己的偏好、项目背景、代码风格等。

**存储什么**：
- 用户偏好（"喜欢简洁代码"、"用中文回复"）
- 项目背景（"这是一个 Web3 学习项目"）
- 过去的决策（"上次选择了 Solidity 0.8+"）
- 反馈（"不要在代码里加注释"）

**我在 Claude Code 中的体验**：
- Claude Code 有 Memory 系统，存在 `.claude/` 目录下
- 比如它记住了我是"AI 有基础、Web3 新手"，后续解释 Web3 概念时会更详细
- 但记忆可能过时，需要定期更新

**风险**：
- 记住错误信息 → 后续对话一直用错误的前提
- 记忆冲突 → 两次对话给了不同的偏好
- 隐私泄露 → 记忆中可能包含敏感信息

---

### MCP（Model Context Protocol）

**是什么**：LLM 与外部工具的统一连接协议。让 AI 能标准化地调用各种外部系统。

**类比**：USB 接口标准。有了 USB，鼠标、键盘、U盘、手机都能用同一种方式连接到电脑。MCP 就是 AI 的"USB"，有了它，数据库、API、文件系统、区块链都能用同一种方式连接到 LLM。

**为什么重要**：
- 没有 MCP：每个工具都需要写专门的适配代码
- 有了 MCP：工具开发者按标准实现一次，所有支持 MCP 的 Agent 都能用

**MCP 的结构**：
```
Agent (LLM)
    ↓ MCP 协议
┌─────────────────────────────────┐
│  MCP Server                     │
│  ├── 数据库工具                  │
│  ├── 文件系统工具                │
│  ├── API 调用工具                │
│  └── 区块链工具                  │
└─────────────────────────────────┘
```

**我在 Claude Code 中的体验**：
- Claude Code 支持 MCP，可以连接外部工具
- 我安装了 Anthropic 的 skills 插件，这就是通过 MCP 扩展能力的例子
- 比如 `frontend-design` skill 让 Claude Code 获得了设计指导能力

**与传统 API 调用的区别**：
| | 传统 API 调用 | MCP |
|--|--------------|-----|
| 连接方式 | 每个 API 单独写代码 | 统一协议，一次实现 |
| 发现能力 | 需要提前知道 API 地址 | Agent 可以自动发现可用工具 |
| 上下文 | 每次调用独立 | 工具结果自动进入 Agent 上下文 |

---

### Skills（技能）

**是什么**：可复用的高层指令集。把复杂的操作封装成一个可调用的"技能"。

**类比**：技能就像菜谱。你不需要每次都从零想怎么做菜，翻开菜谱按步骤来就行。

**与 Tool 的区别**：
- Tool：单个操作（搜索文件、执行命令、调用 API）
- Skill：多个 Tool 的组合 + 决策逻辑（"创建一个前端项目" = 创建目录 + 初始化 + 安装依赖 + 生成模板）

**Skills 的特点**：
- **可复用**：一次定义，多次使用
- **自动发现**：Agent 能根据任务自动选择合适的 Skill
- **动态生成**：可以根据上下文动态调整 Skill 的行为

**我在 Claude Code 中的体验**：
- 安装了 `example-skills` 和 `document-skills` 两个插件
- 比如 `frontend-design` skill 会指导 Claude 按照设计原则生成代码
- `brand-guidelines` skill 会应用 Anthropic 的品牌配色
- Skills 让 Claude Code 从"通用助手"变成"有专业能力的助手"

---

### Tool Calling（工具调用）

**是什么**：模型输出结构化请求，框架执行后回传结果。让 LLM 从"说话"变成"做事"。

**过程**：
```
用户："查一下这个文件有多少行"
  ↓ LLM 分析意图，决定需要调用工具
LLM 输出：{"tool": "bash", "command": "wc -l file.txt"}
  ↓ 框架执行工具
框架返回：{"result": "42 file.txt"}
  ↓ LLM 基于结果生成回答
LLM 输出："这个文件有 42 行"
```

**关键理解**：
- LLM 不直接执行工具，它只是"说出"想调用什么
- 框架（如 Claude Code）负责实际执行
- 执行结果回传给 LLM，LLM 再决定下一步

**我在 Claude Code 中的体验**：
- Claude Code 有 Read、Write、Edit、Bash、Grep、Glob 等工具
- 当我说"帮我看看这个文件"，它会调用 Read 工具
- 当我说"搜索所有 .sol 文件"，它会调用 Glob 工具
- 它会根据任务自动选择合适的工具

**Tool Calling 的安全风险**：
- AI 可能调用错误的工具（比如误调 rm 删文件）
- AI 可能传入错误的参数（比如删错文件）
- 所以需要 Guardrails 来限制 AI 可以调用哪些工具

### Tracing（追踪）

**是什么**：可视化 Agent 的执行链。记录每一步做了什么、调用了什么工具、得到了什么结果。

**类比**：就像程序的调试日志。出了问题可以回溯每一步，找到出错的地方。

**为什么重要**：
- 调试：Agent 出错时，能定位是哪一步出了问题
- 审计：Agent 做了什么操作，有没有越权
- 优化：发现哪些步骤耗时最长，哪些可以优化

**我在 Claude Code 中的体验**：
- 每次 Claude Code 执行操作，都会显示它在做什么（读取文件、编辑代码、执行命令）
- 所有操作都记录在 git commit 历史中
- 如果它改错了代码，可以用 git 回退

**Tracing 记录什么**：
```
[09:00:01] 用户输入："帮我写一个存储合约"
[09:00:02] Agent 决定：调用 Write 工具
[09:00:03] Write: 创建 SimpleStorage.sol
[09:00:04] Agent 决定：调用 Bash 工具
[09:00:05] Bash: git add SimpleStorage.sol
[09:00:06] Bash: git commit -m "添加合约"
[09:00:07] 输出："合约已创建并提交"
```

---

### Guardrails（安全护栏）

**是什么**：输入输出的验证规则。不符合规则的操作会被中止。

**类比**：就像高速公路的护栏。车可以自由行驶，但不能冲出路面。

**Guardrails 的类型**：

| 类型 | 作用 | 例子 |
|------|------|------|
| 输入过滤 | 阻止危险的用户输入 | 过滤 prompt injection |
| 工具限制 | 限制 AI 可调用的工具 | 不允许调用 rm -rf |
| 输出验证 | 检查 AI 的输出是否合规 | 合约代码必须编译通过 |
| 权限控制 | 限制 AI 的操作范围 | 只能操作测试网，不能操作主网 |
| 人工确认 | 关键操作必须人确认 | 签名、转账、合约写入 |

**为什么 AI × Web3 特别需要 Guardrails**：
- 链上操作不可撤销
- AI 可能被 prompt injection 攻击
- 私钥泄露 = 资产丢失
- 合约漏洞 = 资金损失

**我在 Claude Code 中的体验**：
- Claude Code 有权限系统，某些操作需要我确认
- 比如 git push 会问我确认，而不是自动执行
- 我设置了 `.claude/settings.json` 来控制哪些命令可以自动执行

### Handoff（移交）

**是什么**：子任务完成后，将控制权移交给下一个处理者（可能是另一个 Agent、一个工具或人工）。

**类比**：接力赛跑。每个选手跑完自己的段落后，把接力棒交给下一个选手。

**什么时候需要 Handoff**：
- 任务超出当前 Agent 的能力范围
- 需要人工确认关键操作
- 需要调用专门的子 Agent

**例子**：
```
Agent A（代码生成）→ 生成合约代码 → Handoff
Agent B（安全审计）→ 检查漏洞 → Handoff
人工 → 确认无误 → Handoff
Agent C（部署）→ 部署到测试网
```

**我在 Claude Code 中的体验**：
- 当 Claude Code 遇到不确定的操作，会问我确认（Handoff 给人工）
- 比如："这个改动会影响 3 个文件，要继续吗？"
- 这就是 Handoff 的实际应用

---

### 错误恢复（Error Recovery）

**是什么**：执行失败时的重试、回退和人工介入机制。

**类比**：导航软件发现前方封路，自动重新规划路线。

**错误恢复的策略**：

| 策略 | 适用场景 | 例子 |
|------|----------|------|
| 自动重试 | 临时性错误（网络超时） | RPC 调用失败，等 1 秒重试 |
| 回退 | 操作可以撤销 | git reset 回退错误的 commit |
| 换方案 | 原方案不可行 | 用 Hardhat 替代 Remix 部署 |
| 人工介入 | 无法自动解决 | 合约编译失败，问用户怎么修 |
| 放弃并记录 | 任务不值得继续 | 记录失败原因，跳过此步骤 |

**我在 Claude Code 中的体验**：
- 当 Claude Code 执行 bash 命令失败时，会看错误信息，尝试修复
- 比如 git push 失败（网络问题），它会提示我检查网络，而不是直接放弃
- 但它不会无限重试，会停下来问我怎么办

## 7. 什么时候需要 Agent

### 需要 Agent 的信号

| 信号 | 说明 | 例子 |
|------|------|------|
| 目标开放式 | 无法提前写死步骤 | "帮我调查这个 bug" |
| 多工具协作 | 需要调用多个工具 | 搜索代码 + 读日志 + 分析 + 修复 |
| 中间结果决定下一步 | 后续步骤依赖前面的结果 | 读取文件 → 根据内容决定怎么改 |
| 跨会话记忆 | 需要记住之前的对话 | "上次你选择了方案 A" |

### 不需要 Agent 的信号

| 信号 | 说明 | 更好的方案 |
|------|------|-----------|
| 一次性问答 | 只需要一个回答 | 用 Prompt |
| 流程固定 | 步骤可以提前写死 | 用脚本/Workflow |
| 高合规要求 | 不能让 AI 自主决策 | 用人工审核节点 |
| 数据确定性 | 需要精确数据 | 用数据库查询 |
| 成本敏感 | Agent 调用成本高 | 用简单方案 |

### 警惕过度 Agent 化

**复杂度和风险越高，越要警惕过度 Agent 化。**

过度 Agent 化的表现：
- 简单任务用 Agent → 浪费资源，增加出错可能
- 不需要决策的任务让 Agent 自主决策 → 可能做出错误选择
- 关键操作没有人工确认 → 可能造成不可逆损失

**正确的选择**：
```
用户："翻译这段话"
  → 这是简单问答，用 Prompt 就够了
  → 不需要 Agent

用户："帮我部署一个合约到测试网"
  → 需要多步骤：生成代码 → 编译 → 部署 → 验证
  → 但步骤是固定的，用 Workflow
  → 不需要 Agent 自主决策

用户："帮我调查这个项目的安全问题"
  → 目标开放，不知道要检查什么
  → 需要根据中间结果决定下一步
  → 需要 Agent
```

### 我的判断经验

> 在 Week 1 的学习中，我用了 Claude Code（Agent 模式）来完成所有任务。但实际上很多任务不需要 Agent：
> - 生成笔记内容 → Prompt 就够了
> - 生成 HTML 页面 → Prompt 就够了
> - git commit/push → 脚本就够了
> - 但"根据学习计划，自动整理资料、生成笔记、更新 README、提交代码"这个完整流程 → 需要 Agent
>
> 关键是：**Agent 的价值在于串联多个步骤，而不是单个步骤本身。**

## 8. 我的学习笔记

### 我的理解

> LLM 就是一个超级强大的"文字接龙"机器。它不理解语义，但通过海量数据学会了语言的模式。所以它能写出看起来很有道理的话，但可能完全是编的。

### 我的疑问

> - Agent 的"自主决策"边界在哪？什么时候该让人介入？
> - 长期记忆会不会导致 Agent 记住错误信息？
> - MCP 和传统的 API 调用有什么本质区别？

### 我的发现

> 使用 Claude Code 时，提示词越具体，输出质量越高。比如"帮我写一个 Web3 概念测验"不如"帮我写一个 10 道选择题的 Web3 概念测验，深色主题，每题有解释，纯 HTML/CSS/JS"。
